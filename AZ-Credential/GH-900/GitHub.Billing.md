# GitHub Billing and Payment
비용을 발생하지 않고 사용할 수 있지만 별도로 유료 플랜을 구독하거나, 청구된 제품에 대한 플랜에 포함된 일정 할당량을 초과할 경우 사용량에 대한 비용을 지불해야한다. 자세한 청구 및 소비 보고서를 제공하며 이는 보고서를 통해 비용을 모니터링하고, 리소스를 효율적으로 할당함에 조직 정책을 준수할 수 있다.

* GitHub 요금제
  * 유료 GitHub 계정(Pro, Team)이나 Copilot Plan에 대한 고정 월 요금
* 구독
  * 추가 구독 제품에 대한 고정 월 비용(Ex: GitHub Secret Protection)
* 사용량
  * 요금제에 포함된 금액(GitHub Actions, ...etc)에서 할당된 이상으로 특정 기능을 얼마나 사용하는지에 따라 발생하는 변동 비용이다.

## 청구 방식(Billing work)
각 소유한 계정(Personal, Team, Enterprise)별로 별도로 청구하며 각 계정마다 별도의 청구 항목이 있다.

* 청구 날짜
* GitHub 요금제의 청구 기간(1 Month)
* 결제 수단
  * 신용카드/체크카드(Credit Card)
  * PayPal
  * Azure 구독(Azure Subscription)
  * 영수증(Receipt)

### 청구 주기(Billing Cycle)
계정마다 청구 날짜와 주기가 존재한다. 신용카드 및 PayPal 결제용으로 설정된 개인 및 조직 계정 경우, 청구일은 일반적으로 결제 요금제를 시작한날로 부터 설정한 주기에 맞춰 청구 혹은 결제가 진행된다. 하지만 엔터프라이즈 계정 경우 청구 날짜가 경우에 따라 다를 수 있기에 대부분 사용자는 GitHub Settings에서 Biliing and Payment 카테고리에서 청구 내역을 확인할 수 있다.

> 미터 기반 기능 사용 청구 주기는 월 첫날부터 마지막 날까지 고정된 기간이다.

### 보류
구독이나 체험을 시작하여 제공한 결제 수단에 승인 보류가 발생할 수 있다. 은행이나 카드 발급사 사정에 따라 거래나 결제가 발생하거나 보류가 발생하여 일시적이지만 가능한 빠르게 해제되지만, 허가 보류가 영업일 10일 이상 지속되면 해당 은행이나 카드사에 문의 해야한다. 승인 실패로 인해 계정이 잠금(Block)상태로 지원팀에 문의해야 한다.

## 주요 기능(Key Functionalities)
청구 및 라이선스 페이지에는 유료 제품, 요금제, 구독 사용 비용을 이해하고 관리하는 데 도움이 되는 뷰와 옵션이 포함되어 수행할 수 있는 업무는 다음과 같다.

* 과도한 지출 방지(`Prevent Overspending`)
  * 예산과 알림을 활용해 지출을 추적하고 통제
* 지출 관찰 및 이해(`Observe and understand spending`)
  * 지출이 제품 전반에 어떤 과정으로 분배되는지 이해
* 미래 지출 추정(`Estimate future spending`)
  * 비용 센터(엔터프라이즈)와 예산(All)에서의 사용량을 기반으로 지출 추세를 확인
* 인사이트와 데이터 수집(`Gather insights and data`)
  * 팀이나 이해관계자와 공유할 사용 보고서를 생성하고, 예산이 정확한지 확인
* 센터에 비용 배분(`Allocate costs to centers`)
  * 조직, 저장소, 사용자를 비용 센터에 배치하여 책임성 향상

## 조직의 가격 챙정 및 지원 옵션
조직의 요구 사항에 맞게 조정된 다양한 플랜을 제공하며 각 고유한 기능과 지원 옵션을 제공한다.

* 조직 :
  * 무한한 Public / Private 리포지토리
  * 커뮤니티 지원
  * 매월 2,000분 CI/CD

  * 추가 공동 작업 도구
  * 코드 소유자
  * 필수 검토

  * 고급 감사
  * GitHub Connect
    * 엔터프라이즈 수준 보안 기능

## 라이선스에 대한 사용량 기반 청구
향상된 청구 플랫품을 통해 GitHub는 라이선스에 대한 사용량 기반 청구 모델은 도입했다.

* 월별 청구
    * 조직은 사용된 GitHub Enterprise 및 GitHub Advanced Security 라이선스의 정확한 수에 대해 매월 청구된다.
* 비레 요금
    * 사용자가 한 달 중반에 라이선스를 사용하기 시작하면 조직은 해당 사용자의 사용량에 대해 비례 금액이 청구된다.
* 동적 조정
    * 사용자가 한 달 동안 라이선스 사용을 중지하는 경우 다음 달의 청구는 이러한 변경 사항을 반영하여 조직에서 활성 사용자에 대해서만 비용을 지불

## 청구 플랫품의 기능
* 세분화된 지출 제어
  * 관리자는 GitHub Actions 및 Package와 같은 서비스에 대한 특정 지출 한도를 설정하여 의도치 않은 요금 초과를 방지할 수 있다.
* 자세한 사용 현황 인사이트
  * 플랫품은 제품 사용량에 대한 심층적인 가시성을 제공하여 조직이 소비 패턴을 모니터링하고 리소스 할당을 최적화할 수 있도록 한다.
* 자동화 보고
  * 사용량 보고를 자동화하는 기능은 재무 감독을 간소화하고 내부 비용 회수를 용이하게 도와준다.

### 추가 고려 사항
* GitHub CodeSpace
  *  계획에 따라 스토리지 및 컴퓨팅 시간에 따라 추가 요금이 발생한다.
* 청구 경고
  * 관리자는 GitHub UI 및 이메일 알림을 통해 지출 한도에 근접할 때 경고 안내를 받을 수 있다.


# 측정된 사용량 보고서
GitHub는 요금제 제품의 사용량을 추적하기 위한 자세한 청구 및 소비 보고서를 제공한다. 이는 보고서를 통해 관리자는 비용을 모니터링하고, 리소스를 효율적으로 할당하며 조직 정책을 준수할 수 있다.

## GitHub Package Storage
아티팩트, 컨테이너 이미지 및 종속성을 저장할 수 있다. 스토리지는 저장된 데이터 불륨 및 데이터 전송 사용량에 따라 측정된다.

### 소비 추적
* 스토리지 사용량을 확인하고자 한다면 설정 -> 청구 -> 패키지로 이동
* 분석에는 리포지토리당 사용되는 스토리지(GB) 및 데이터 전송(GB)가 포함된다.

### 청구 세부 정보
* 무료 할당
  * 공용 리포지토리(무료 스토리지 및 대역폭 제한)
  * 프라이빗 리포지토리
    * 최대 2GB의 스토리지
    * 매월 최대 1GB의 데이터 전송

### 최적화 전략
* 사용하지 않는 패키지를 정기적으로 정리하거나 보존 정책을 사용하도록 설정
* 액세스 빈도가 잦은 이미지를 중앙 레지스트리에 저장하여 중복률을 감소
* 압축 형식을 사용하여 스토리지 사용량을 최적화

## GHE 라이선스
GitHub Enterprise는 조직을 위한 고급 기능을 제공하며 활성 사용자 수가 라이선스 사용을 결정한다.

### 소비 추적
* 엔터프라이즈 설정 -> 설정으로 이동하여 라이선스 사용 현황 보고서를 확인
* 활성 사용자와 할당된 라이선스를 모니터링 진행

### 청구 세부 정보
* 가격 책정 모델
  * 프라이빗 리포지토리에 액세스할 수 있는 각 사용자는 하나의 라이선스를 사용
  * 조직은 매년 혹은 매월 사용자당 요금을 지불
* 비활성 사용자
  * 관리자가 사용자를 제거하는 경우 라이선스는 청구 기간 동안 활당된 상태로 유지되지만 다시 할당될 수 있다.

### 최적화 전략
* 비활성 사용자를 감사하고 액세스 권한을 취소하여 라이선스를 확보
* SSO 및 SCIM 프로비저닝을 사용하여 사용자 관리를 자동화


## GitHub Advanced Security(GHAS) License
`GHAS`는 향상된 보안을 위해 코드 검사 및 종속성 검토를 제공

### 소비 추적
* 사용량을 확인하고자 한다면 GitHub.com 탐색에서 Enterprise를 선택하고 목록에서 엔터프라이즈를 서낵한 다음, 청구 -> 고급 보안으로 이동
* 활성 커밋은 커밋이 처음 작성된 시기에 관계없이 지난 90일 이내에 사용하도록 설정된 GHAS를 사용하여 리포지토리에 하나 이상의 커밋을 푸시한 사용자
* GHAS 청구는 청구 기간당 고유한 활성 커밋자 수를 기반으로 책정

### 청구 세부 정보
* 가격 책정 모델
  * 월별 고유 커밋자 당 요금이 청구
  * 커밋자가 여러 리포지토리에 기여하는 경우 한 번만 계산
* 무료 계층
  * 공용 리포지토리에만 사용 불가

### 최족화 전략
* 고급 보안이 진정으로 필요한 리포지토리로 GHAS를 제한
* 분기 보호를 사용하여 기능 분기에서 불필요한 검사를 제한

## GitHub Copilot
사용자당 청구 되는 AI 기반 코드 오나성 및 제안을 제공

### 소비 추적
* 관리자는 조직 설정의 청구 -> Copliot 사용량을 추적 가능
* 보고서에는 활성 사용자 및 월별 청구 예상 금액이 표시

### 청구 세부 정보
* 엑세스 모델
  * 구독 옵션이 다른 개인 및 비즈니스에 사용 가능
* 무료 액세스
  * 학생 및 확인된 오픈 소스 유지 관리자에게는 무료
  * 특정 엔터프라이즈 고객을 위한 평가판 무료

### 최적화 전략
* 코필로트가 필요하지 않은 사용자를 위해 정기적으로 Copilot를 검토하고 비활성화 처리
* 개발자가 AI 생성 코드가 필요하지 않은 프로젝트에서 Copilot를 사용하지 않도록 설정 하도록 권장

## Lage File Storage(LFS)
GitHub LFS는 Git 리포지토리와 별도로 큰 이진 파일을 저장하는 데 사용

### 소비 추적
* 청구 -> LFS 사용량에서 LFS 사용량을 추적
* 보고서에는 스토리지(GB) 및 대역폭 사용량(GB)이 포함

### 청구 세부 정보
* 무료 계층
  * 매월 계정당 1GB의 스토리지와 1GB 대역폭 사용량

### 최적화 전략
* 대용량 파일에 외부 스토리지 서비스를 사용
* 사용되지 않는 대용량 파일을 정리하여 스토리지의 공간 확보 및 최적화
* Git LFS 파일 정리를 사용하도록 설정하여 참조되지 않은 개체를 제거
