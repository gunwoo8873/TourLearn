# GitHub Builing
개인 및 조직 혹은 엔터프라이즈 유형에 따라 비용을 계정에 청구한다. 개인과 조직은 별도의 청구서를 받지만 엔터프라이즈는 여러 조직에서 청구를 통합하거나 내부 구성에 따라 다른 비용 센터로 요금을 라우팅할 수 있다. 각 청구서는 구독 및 사용량 기반 요금의 조합을 반영한다.

* 구독은 계정의 계획 GitHub Copliot과 Marketplace 앱과 같은 확정성이 있는 제품의 고정 월별 비용이 포함된다.
* 사용량 기반 청구는 GitHub Action나 Package와 같은 성격의 기능을 사용 시 비용이 청구되는 제품에 적용된다.

## GitHub Actions 가격 책정
리포지토리 내에서 직업 워크플로를 자동화할 수 있으며, 가격 책정 모델은 리포지토리 표시 유형 및 계정 유형에 따라 달라진다.

* Public
  * 공룡 리포지토리에 대해 무료로 제공되며 GitHub 호스팅 실행기에서 무제한 시간을 제공한다.
* Private
  * 각 계정은 계획에 따라 GitHub 호스팅 실행기를 위한 주어진 무료 시간과 스토리지를 제공하여 플랜에 해당하는 제한적인 사용 범위가 존재한다.

## 조직의 가격 챙정 및 지원 옵션
조직의 요구 사항에 맞게 조정된 다양한 플랜을 제공하며 각 고유한 기능과 지원 옵션을 제공한다.

* 조직 :
    * 무한한 Public / Private 리포지토리
    * 커뮤니티 지원
    * 매월 2,000분 CI/CD
*  팀 :
    * 추가 공동 작업 도구
    * 코드 소유자
    * 필수 검토
    * 강제 분기 보호
    * 전자 메일 지원
* 엔터프라이즈 :
    * SAML(보안 인증 마크업 언어) 단일 로그인
    * 고급 감사
    * GitHub Connect
      * 엔터프라이즈 수준 보안 기능

* 조직은 공동 작업 요구 사항 및 보안 우선 순위를 평가하여 목표에 가장 적합한 계획을 선택해야 합니다.

## 라이선스에 대한 사용량 기반 청구
향상된 청구 플랫품을 통해 GitHub는 라이선스에 대한 사용량 기반 청구 모델은 도입했다.

* 월별 청구
    * 조직은 사용된 GitHub Enterprise 및 GitHub Advanced Security 라이선스의 정확한 수에 대해 매월 청구된다.
* 비레 요금
    * 사용자가 한 달 중반에 라이선스를 사용하기 시작하면 조직은 해당 사용자의 사용량에 대해 비례 금액이 청구된다.
* 동적 조정
    * 사용자가 한 달 동안 라이선스 사용을 중지하는 경우 다음 달의 청구는 이러한 변경 사항을 반영하여 조직에서 활성 사용자에 대해서만 비용을 지불

## 청구 플랫품의 기능
* 세분화된 지출 제어
  * 관리자는 GitHub Actions 및 Package와 같은 서비스에 대한 특정 지출 한도를 설정하여 의도치 않은 요금 초과를 방지할 수 있다.
* 자세한 사용 현황 인사이트
  * 플랫품은 제품 사용량에 대한 심층적인 가시성을 제공하여 조직이 소비 패턴을 모니터링하고 리소스 할당을 최적화할 수 있도록 한다.
* 자동화 보고
  * 사용량 보고를 자동화하는 기능은 재무 감독을 간소화하고 내부 비용 회수를 용이하게 도와준다.

### 추가 고려 사항
* GitHub CodeSpace
  *  계획에 따라 스토리지 및 컴퓨팅 시간에 따라 추가 요금이 발생한다.
* 청구 경고
  * 관리자는 GitHub UI 및 이메일 알림을 통해 지출 한도에 근접할 때 경고 안내를 받을 수 있다.

## 컴퓨터 및 주변 장치에서 라이선스 사용량 통계

### 컴퓨터 계정 및 주변 장치 서비스 이해
자동화, 스크립트 실행 또는 타사 도구와 통합하는 데 사용되는 GitHub 계정이다. 사용자와 독립적으로 작동하며 CI/CD에서 자주 사용되기에 각 컴퓨터 계정은 표준 사용자와 같은 GitHub 라이선스를 사용한다.

### 주변 장치 서비스
주변 장치 서비스는 API 요청을 통해 GitHub와 상호 작용하는 외부 통합이다.
그렇다면 추적을 왜 하는가에 대한 이유는 다음과 같다.

* 사용되지 않거나 과도한 라이선스를 식별
* 비용을 최적화하고 불필요한 지출을 방지
* 비활성 또는 잘못 구성된 자동화 계정의 보안 리스크를 모니터링

## 컴퓨터 계정 및 주변 서비스 라이선스 관리 모범 사례
* 정기적 계정 감사
  * 시간에 지남에 따라 조직은 리포지토리 및 시스템에 계속 액세스할 수 있는 사용되지 않거나 부실한 계정을 누적한다. 이는 비활성 계정의 손상으로 악용 리스크가 발생하여 보안 위험을 초래할 수 있기에 정기적인 감사는 활성 및 필요한 계정만 존재하도록 하여 무단 액세스에 대한 노출을 줄인다.
* API 사용량 모니터링
  * 많은 타사 애플리케이션, CI/CD 파이프라인 및 통합은 GitHub API 리소스 및 엔터프라이즈 라이선스를 사용한다 과더한 API 호출은 속도 제한으로 이어져 개발자의 생산성에 영향을 미칠 수 있기에 무단 혹은 식별할 수 없는 API 사용은 중요한 데이터 및 보안 취약성을 노출할 수 있다.
* 실행기 사용 최적화
  * 자체 호스팅 및 GitHub 호스팅 실행기는 CI/CD 워크플로를 실행하여 비효율적인 사용으로 불필요한 비용이 발생하는 사례가 존재 유휴 상태의 자체 호스팅 실행기는 컴퓨팅 리소스를 낭비하고 모니터링되지 않은 상태로 두면 조직이 보안 위험에 노출된다. 또한 사용한 만큼 지불하는 방식으로 운영되기에 최적화를 지속적으로 개선하지 않으면 비용을 줄일 수 없다는 문제가 존재한다.
* 사용자 계정 제한
  * 사용자 계정에는 리포지토리에 대한 불필요한 액세스 권한이 없어야 하므로 권한 상승의 위험이 줄어든다 손상된 경우 악용하여 소스 코드를 조작하거나, 악의적인 변경 내용을 배포하거나, 민감한 보안 내용들을 노출할 수 있기에 보안 정책을 적용하여 규정 준수를 보장하고 잠재적인 위반을 최소화할 수 있다.
